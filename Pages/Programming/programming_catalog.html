Autastic Education - Companion Application

The Autastic project consists of two primary applications, the therapy application (found under games) and this application. This is the companion app meant to be used by parents and clinical professionals. For the companion application I have had a primarily managerial role focusing largely on design and architecture.

This application provides a unified location for parents and clinicians to digest and utilize the statistics and progress information generated by the therapy software. It furthermore allows these parents and professionals to configure and interact with the therapy software to maximize the therapy outcomes of their children or patients.

The Autastic project is an ongoing project. As such the images here are from a development build and do not necessarily reflect the finished product.

Technical Details and Notes

The Autastic Companion Application is developed in c# .net. It utilizes .net MAUI to create a multi platform application user interface. Its functionality is made possible by Microsoft Entra ID, Cosmos DB and other cloud services and technologies.

UVSim IDE - Basic Machine Language Assembler and Interpreter

This application represents the culmination of a project I made for a Software Engineering class. It begins at its core with a very simple assembler that converts a rudimentary assembly language that was designed to demonstrate machine code by the use of instruction codes as assembly mnemonics. The assembler creates a RISK style intermediate byte-code that is then digested by a runtime interpreter.

After the fundamental assembler and interpreter were implemented, I decided to create a GUI that would allow for the editing of the "Basic Machine Language" programs. As well as this editor we were tasked to create some kind of GUI that would allow the monitoring of the interpreters resources. In the end what I ended up making was an Integrated Development Environment that allows the creation of "Basic Machine Language" programs, the ability to then assemble those programs into the intermediate binary, and finally execute those programs on the runtime interpreter and view the systems simulated resources in a graphical interface, all from within one integrated environment. Hence the UVSim IDE.

Technical Details and Notes

The entire project was built using c# .net. In the end I would probably not build the entire project in c# if I were to do it over again. The decision to use c# was made out of a desire to use .net MAUI for the eventual GUI. This initially proved to be a bit headache as MAUI originally did not support building directly to a standalone executable on windows. However, this support has since been added and I have been able to go back and rebuild to a standalone executable.

This project was developed shortly after .net MAUI was officially released, and suffice it to say its support for desktop GUI development seemed to be a bit of an afterthought at the time. Not only were the MAUI visual elements more oriented for mobile UIs, at the time that I built this project MAUI could not be used to build to an exe. I did not know this when I began, and only became aware at the very end of development.

All said and done, UVSim IDE is definitely not perfect, but it was a great learning experience. In particular it was a unique opportunity to get a head start on thinking about certain aspects of assembler design and implementation that I would use soon thereafter when beginning the CS capstone. These realizations when elaborated on and incorporated into my assembler, virtual machine, and compiler made the development of those pieces of software much easier, more rapid, and resulted in a much better final product. Furthermore, it has allowed the code of my capstone items to be much more receptive to changes and even major feature additions.


For those curious about some of the design elements, here is a class diagram I made for the UVSim model. Sorry, I know the UML formatting on this one is not great.

The most important aspect of the UVSim design - the one that was ultimately carried over into and expanded upon in the capstone - is the idea of a hybrid command and visitor pattern in the assembler and virtual machine's instruction handling. The idea is to create independent instruction objects for the respective software that become a piece of a larger machine that digests the input and uses those smaller pieces to get the job done.

Card Flip Memory Game

This application is a relatively simple memory game made in an effort to learn some ways to make GUI by hand in a c++ program. The application creates a grid with a randomly shuffled deck of cards upon startup. The user can then select the face down cards one at a time to flip them face up and reveal the identity of the card. As the user finds and selects matches, those matched cards will remain face up together until either all four of that cards kind are selected in sequence, or the sequence is broken by the selection of a different kind of card, at which time the four of a kind cards will be removed from the field.

All in all the project was a good learning opportunity. Not only creating the c++ objects and functions needed to run the game, but also to become a little familiar with the SFML cross platform GUI libraries and development of c++ applications on mac with xcode, which is not my normal development environment or target platform.

Technical Details and Notes

The entire project was built using c++ and the SFML GUI libraries. Given that I now have more experience with at least a few other GUI options, I can say I prefer a more modern approach. At least in 2019 when I built this, all the GUI related elements were hard coded compared to how newer frameworks like MAUI or WIN UI allow a more visual markup approach. That being said, cross platform is very nice and unlike MAUI, SFML would let me build to an exe.

Image Rotator

This application was programmed in c# and the GUI was done with MAUI. The user can load in any number of images, and then selectively rotate the images. After rotating images the user can save edits over the original image files, or create new image files.

This project was a great opportunity to develope a Graphical User Interface that provides a functional and appealing user experience. I also enjoyed learning how to use some .net graphical libraries to manipulate images and then serialize those changes.

Technical Details and Notes

The project was built using c# .net. The logic for the program utilizes some simple modular services and data models to modify the images. The GUI utilizes the MVVM design pattern to create the visual elements and interface the visual items with the backend of the program.

The backend (model) is composed of a few small modules that define the structure of the data needed by the program, and the logic used to manipulate that data so that the images can be rotated. It also defines an interface that allows for dependency injection of a file picker on a specific platform as at the time of development .net and MAUI didn't have a unified file picker that was cross platform.

The frontend (View in the MVVM design pattern) is connected to the logic of the program via the view-models. The view-models contain references to instances of items from the model and knows how and when to call logic from the backend to manipulate the models.
